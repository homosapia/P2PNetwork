# P2P Network Core — план разработки децентрализованного сервиса

Децентрализованный P2P сервис с упором на безопасность, устойчивость к отказам и автономность узлов.

---

## 🧱 Фаза 1: Фундамент сети

### 1.1 Идентификация ноды
- Детерминированная генерация криптографической пары ключей (Ed25519) из seed-фразы
- Сохранение идентичности между перезапусками засчет seed-фразы
- `NodeId` случайно сгенерированное имя для DHT Kademlia.
- Создание объекта данных узла для обмена и подписание этих данных закрытым ключом.

### 1.2 Обнаружение нод (Peer Discovery)
- DNS Bootstrap через seed-ноды
- Сохранение списка пиров на диск
- Health-check известных пиров при запуске
- API для обмена списками пиров

### 1.3 Базовая коммуникация
- Ping/Pong для проверки доступности
- Обмен информацией о пирах
- REST API для взаимодействия
- Таймауты и retry-логика

> ✅ **Результат фазы:** Ноды находят друг друга и обмениваются списками пиров.

---

## 🔁 Фаза 2: Устойчивость сети

### 2.1 Мониторинг сети
- Периодическая проверка известных пиров
- Удаление мёртвых нод из таблицы
- Обнаружение новых нод через gossip
- Метрики здоровья сети (активные ноды, latency)

### 2.2 Постоянное хранилище пиров
- Сохранение пиров между перезапусками
- Приоритезация стабильных нод
- Очистка устаревших записей
- Экспорт/импорт списка пиров

### 2.3 Переподключение
- Автоматический поиск новых пиров при падении соединений
- Поддержание минимального количества соединений
- Back-off стратегия при неудачных попытках
- Разделение на inbound/outbound соединения

> ✅ **Результат фазы:** Сеть самовосстанавливается при отключении нод.

---

## 🔐 Фаза 3: Безопасность

### 3.1 Аутентификация «свой-чужой»
- Подпись сообщений приватным ключом
- Верификация подписи публичным ключом
- Привязка `NodeId` к публичному ключу
- Защита от подмены идентичности

### 3.2 Доверенные пиры
- White-list доверенных `NodeId` для seed-нод
- Trust-On-First-Use для новых пиров
- Обнаружение смены ключей (alert)
- Чёрный список заблокированных нод

### 3.3 Защита от атак
- Rate limiting на API
- Защита от Sybil-атак (ограничение нод с одного IP)
- Валидация входящих данных
- Защита от Eclipse-атаки (множественные seed-ноды)

> ✅ **Результат фазы:** Только авторизованные ноды участвуют в сети.

---

## 📡 Фаза 4: Синхронизация данных

### 4.1 Протокол синхронизации
- Обнаружение расхождений данных
- Сравнение версий (векторные часы или Merkle-деревья)
- Дельта-синхронизация (только изменения)
- Разрешение конфликтов (CRDT или last-write-wins)

### 4.2 Репликация
- Выбор нод для репликации (по расстоянию `NodeId`)
- Стратегия хранения (сколько копий данных)
- Восстановление данных при падении ноды
- Гео-распределённая репликация (опционально)

### 4.3 Консистентность
- Модель: `eventual consistency`
- Подтверждение записи от кворума нод
- Обработка сетевых разделений (split-brain)
- Восстановление после разделения

> ✅ **Результат фазы:** Данные синхронизированы между нодами.

## 🚀 Фаза 5: Продвинутые возможности

### 5.1 DHT (опционально)
- Реализация Kademlia-подобной таблицы
- XOR-метрика для поиска
- Маршрутизация запросов
- Распределённое хранение данных

### 5.2 NAT Traversal
- Обнаружение NAT-типа
- UDP hole punching
- STUN/TURN интеграция
- Поддержка работы за NAT

### 5.3 Шифрование канала
- End-to-end шифрование сообщений
- Perfect Forward Secrecy
- Защита от man-in-the-middle
- Шифрование хранимых данных

> ✅ **Результат фазы:** Полнофункциональная защищённая P2P сеть.
